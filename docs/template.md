# Template Guide

This repository is a starter template for Python projects. It provides a
package layout, tests, linting, type checking, documentation, Docker support,
and GitHub Actions workflows that are ready to use after bootstrapping.

## Create a project

Create a repository from this template with GitHub's **Use this template**
button, or clone it directly if you want to experiment locally.

After creating a real project, run the bootstrap script if the GitHub workflow
did not already do it:

```bash
python scripts/bootstrap_template.py your-repository-name
```

The script updates project metadata, rewrites package imports, renames the
source package directory, and adjusts CI settings. Repository names using
dashes are valid; the Python package name defaults to the same name with dashes
converted to underscores.

Use these options when the inferred values are not what you want:

```bash
python scripts/bootstrap_template.py your-repository-name \
  --package-name your_package \
  --project-title "Your Project" \
  --author "Your Name" \
  --author-email "you@example.com" \
  --description "Short project description." \
  --minimum-python-version 3.12
```

## Repository layout

The source package lives in `project_name/`. Tests live in `tests/`, examples
live in `examples/`, documentation lives in `docs/`, and automation scripts live
in `scripts/`.

Keep the public API documented with Google-style docstrings. The documentation
build imports the package and renders those docstrings into the API reference.

## Development workflow

Install the development dependencies:

```bash
uv sync --extra dev --extra docs
```

Run the main checks locally:

```bash
uv run pytest
uv run tox -e type
uv run tox -e coverage
uv run tox -e security
uv run tox -e build
uv run tox -e docs
```

Run the generated-project smoke test after changing bootstrap, packaging, CI,
or documentation infrastructure:

```bash
RUN_TEMPLATE_SMOKE=1 uv run pytest tests/test_template_smoke.py
```

Use the formatter environment when you want Ruff to apply fixes:

```bash
uv run tox -e formatter
```

## Documentation workflow

Manual documentation pages are Markdown files parsed by MyST. API documentation
is generated by Sphinx from the Python package and its Google-style docstrings.

Build the docs exactly as CI does:

```bash
uv run tox -e docs
```

This writes the HTML site to `docs/_build/html` and fails on Sphinx warnings.
Run a live preview while writing:

```bash
uv run sphinx-autobuild docs docs/_build/html
```

See `Building Documentation` for the full build, CI, and Read the Docs flow.

## Continuous integration

The `Test and build` workflow runs pre-commit checks, tests the package across
the configured Python versions, runs type checks on the minimum supported
Python version, enforces coverage, audits dependencies, validates package
artifacts, builds the Docker image, and builds the documentation as a strict
Sphinx build.

The `Bootstrap Template` workflow runs when a repository is created from the
template. It rewrites placeholders automatically and commits the result back to
the new repository when GitHub provides enough metadata.

## Publishing documentation

The repository includes a Read the Docs configuration. Import the repository in
Read the Docs and it will build the Sphinx site from `docs/conf.py`.

GitHub Actions also validates the documentation on pull requests so broken API
references or malformed Markdown are caught before publishing.
